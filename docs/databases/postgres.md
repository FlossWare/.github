# PostgreSQL Architecture

PostgreSQL with the pgvector extension serves as the primary data store for the
distributed LLM orchestration framework. It consolidates relational data, vector
similarity search, and analytical queries into a single ACID-compliant system,
eliminating the operational burden of synchronizing multiple purpose-built
databases.

**Version:** PostgreSQL 16 with pgvector 0.7+
**Location:** Controller node (aio-01), port 5433
**Database name:** `learning`

---

## Schema Overview

The database contains 226 tables organized across 25 schemas. Each schema owns
a bounded domain of the system's concerns.

```
learning (database)
  |
  +-- knowledge.*    (20 tables)   Knowledge base and document store
  +-- learning.*     (85+ tables)  Model capabilities, strategies, bandit state
  +-- monitoring.*   (30+ tables)  Fleet health, API status, execution logs
  +-- workflow.*     (17 tables)   Multi-AI workflow tracking
  +-- costs.*        (8 tables)    Token and dollar cost accounting
  +-- ga.*           (12 tables)   Genetic algorithm populations and fitness
  +-- experiments.*  (10 tables)   A/B tests and controlled experiments
  +-- fleet.*        (9 tables)    Node inventory, capabilities, assignments
  +-- evaluation.*   (7 tables)    Adversarial evaluation harness results
  +-- ... (16 additional schemas for ancillary concerns)
```

## Key Schemas

### knowledge.* (20 tables)

Stores the ingested document corpus and its derived artifacts.

| Table                 | Purpose                                        |
|-----------------------|------------------------------------------------|
| documents             | Raw document metadata (URL, title, source)     |
| chunks                | Text segments with positional offsets           |
| embeddings            | 768-dim vectors for semantic search             |
| verification_votes    | Multi-model consensus on factual claims         |
| provenance            | Source lineage (which scraper, when, hash)       |
| concepts              | Extracted named entities and topics              |
| relationships         | Directed edges between concepts                  |

The `embeddings` table holds 768-dimensional vectors generated by
`sentence-transformers/all-MiniLM-L6-v2`. An HNSW index on this column supports
approximate nearest-neighbor queries in O(log n) time.

```sql
-- Find the 10 most similar chunks to an input vector
SELECT c.content, e.embedding <=> $1::vector AS distance
FROM knowledge.embeddings e
JOIN knowledge.chunks c ON c.id = e.chunk_id
ORDER BY distance
LIMIT 10;
```

### learning.* (85+ tables)

The largest schema family. Tracks everything the system learns about model
behavior, strategy effectiveness, and bandit state.

| Table Group           | Count | Purpose                                    |
|-----------------------|-------|--------------------------------------------|
| model_capabilities    | 1     | Per-model task-type proficiency scores      |
| strategy_performance  | 1     | Thompson Sampling alpha/beta state          |
| bandit_models         | 4     | UCB1, Exp3, LinUCB, Thompson Sampling       |
| ga_configs            | 6     | GA population snapshots and fitness history  |
| experiences           | 1     | 128-dim experience vectors for recall        |
| memories              | 1     | Extracted learnings from past executions      |
| (remaining ~70)       | --    | Supporting indices, partitions, audit logs    |

### monitoring.* (30+ tables)

Operational telemetry for the fleet and external API providers.

| Table                    | Purpose                                         |
|--------------------------|-------------------------------------------------|
| api_health_status        | Per-provider availability and response times     |
| execution_summary        | Per-request model, tokens, latency, outcome      |
| fleet_health             | Node CPU, memory, disk, last heartbeat           |
| rate_limits              | Current rate-limit counters per provider          |
| network_latency_matrix   | Pairwise latency between fleet nodes              |
| diversity_alerts         | Fired when model usage skews beyond thresholds    |

### workflow.* (17 tables)

Tracks multi-AI workflow executions end to end.

| Table              | Purpose                                           |
|--------------------|---------------------------------------------------|
| executions         | Top-level workflow run (name, status, duration)    |
| worker_results     | Individual worker outputs per execution            |
| arbiter_decisions  | Arbiter synthesis and final verdicts               |
| phases             | Phase-level timing and status                      |
| feedback           | Post-execution quality ratings                     |
| learnings          | Extracted insights for future runs                 |
| consensus_cache    | Cached consensus results to avoid redundant calls  |
| ab_test_*          | A/B test configuration and variant assignments     |

## Vector Dimensions and Indexing

The system uses three vector dimensionalities, each tuned to its domain:

```
+---------------------+------+---------------------+---------------------------+
| Domain              | Dims | Index Type          | Rationale                 |
+---------------------+------+---------------------+---------------------------+
| experiences         |  128 | HNSW (m=16, ef=64)  | Fast recall, small corpus |
| workflow embeddings |  384 | HNSW (m=16, ef=100) | Moderate corpus, balanced |
| knowledge chunks    |  768 | HNSW (m=32, ef=200) | Large corpus, high recall |
+---------------------+------+---------------------+---------------------------+
```

HNSW (Hierarchical Navigable Small World) provides O(log n) approximate
nearest-neighbor search. The `m` parameter controls graph connectivity (higher =
more accurate but more memory), and `ef_construction` controls build-time
quality.

```sql
-- Create an HNSW index on knowledge embeddings
CREATE INDEX idx_knowledge_embeddings_hnsw
ON knowledge.embeddings
USING hnsw (embedding vector_cosine_ops)
WITH (m = 32, ef_construction = 200);
```

## Performance Benchmarks

Comparative benchmarks against ChromaDB, measured on the same hardware with
identical datasets:

```
Operation                    ChromaDB    pgvector    Speedup
---------------------------------------------------------------
Simple similarity (128-dim)  0.90 ms     0.44 ms     2.0x
Filtered similarity          2.30 ms     0.44 ms     5.2x
768-dim similarity           1.20 ms     0.97 ms     1.2x
Batch insert (1K vectors)    45 ms       38 ms       1.2x
---------------------------------------------------------------
```

The filtered similarity benchmark is the most significant. ChromaDB must
retrieve candidates and then filter, while pgvector pushes predicates into the
index scan. As filter selectivity increases, the gap widens.

## Why PostgreSQL Over Dedicated Vector Databases

A purpose-built vector database (Pinecone, Weaviate, Milvus, ChromaDB) would
handle similarity search, but not the other 90% of the system's data needs.

```
+------------------------------+----------+--------+
| Capability                   | pgvector | Vector |
|                              |          | DB     |
+------------------------------+----------+--------+
| Vector similarity search     |    Y     |   Y    |
| ACID transactions            |    Y     |   N*   |
| Complex JOIN queries         |    Y     |   N    |
| Aggregation / analytics      |    Y     |   N    |
| Foreign keys / constraints   |    Y     |   N    |
| Materialized views           |    Y     |   N    |
| Backup / replication tooling |    Y     |   Y    |
| Schema migrations            |    Y     |   N    |
+------------------------------+----------+--------+
* Some vector DBs offer limited transactional guarantees.
```

The critical advantage is combining vector search with relational predicates in
a single query. For example, finding similar documents filtered by source,
recency, and verification status requires no cross-system coordination:

```sql
SELECT d.title, e.embedding <=> $1::vector AS distance
FROM knowledge.embeddings e
JOIN knowledge.documents d ON d.id = e.document_id
WHERE d.source = 'openwrt'
  AND d.created_at > NOW() - INTERVAL '30 days'
  AND d.verified = true
ORDER BY distance
LIMIT 10;
```

## Access Pattern: REST API Gateway

No component connects to PostgreSQL directly. All access flows through the
REST API gateway on the controller node (port 5000), which provides:

- Connection pooling and query parameterization
- Authentication and rate limiting
- Schema-aware request validation
- Automatic retry with backoff on transient failures

```
+-----------+     +-------------+     +------------+
| Worker    | --> | REST API    | --> | PostgreSQL |
| Nodes     |     | (port 5000) |     | (port 5433)|
+-----------+     +-------------+     +------------+
                        ^
+-----------+           |
| Workflows | ----------+
+-----------+
```

Application-level adapters (`shared/postgres-adapter.js` for JavaScript,
`postgres_adapter.py` for Python) wrap the REST API with typed methods:

```javascript
const { getDB } = require('./shared/postgres-adapter.js');
const db = getDB();
const rows = await db.query(
  'SELECT * FROM learning.strategy_performance ORDER BY avg_reward DESC'
);
```

## Schema Migrations

22 migration files manage schema evolution. Migrations are sequential,
idempotent, and applied through the REST API's `/admin/migrate` endpoint.

```
migrations/
  001_initial_learning_schema.sql
  002_add_knowledge_tables.sql
  003_add_workflow_tracking.sql
  ...
  022_add_ga_convergence_tracking.sql
```

Each migration is wrapped in a transaction. Failed migrations roll back
cleanly, and the `schema_migrations` table tracks which have been applied.

## Materialized Views

Three materialized views provide pre-aggregated analytics and refresh
automatically every 5 minutes:

| View                        | Source Tables             | Purpose                       |
|-----------------------------|---------------------------|-------------------------------|
| workflow.workflow_summary   | executions, worker_results| Aggregated success rates      |
| workflow.model_performance  | worker_results, feedback  | Per-model quality metrics     |
| workflow.cost_analysis      | executions, costs.entries | Cost breakdowns by workflow   |

```sql
-- Refresh is handled by pg_cron, but can be triggered manually
REFRESH MATERIALIZED VIEW CONCURRENTLY workflow.workflow_summary;
```

The `CONCURRENTLY` keyword allows reads during refresh. Without it, the view
would be locked for the duration of the rebuild.

## Backup and Recovery

- **Schedule:** Daily at 02:00 UTC via cron
- **Method:** `pg_dump` with custom format (`-Fc`) for parallel restore
- **Destination:** NFS mount on server-ap
- **Retention:** 30 days, oldest pruned automatically
- **Point-in-time recovery:** WAL archiving enabled for sub-daily granularity
